# Roadmap — Развитие проекта Chess

> Фичи отсортированы по приоритету и зависимостям.
> Оценка трудозатрат: **S** (1-2 часа), **M** (3-5 часов), **L** (1-2 дня), **XL** (3-5 дней)

---

## Фаза 1: Стабилизация и базовые улучшения

### 1.1 Выбор контроля времени
- **Статус:** Реализовано
- **Описание:** Позволить игрокам выбирать time control перед игрой: Bullet (1+0, 2+1), Blitz (3+0, 3+2, 5+0, 5+3), Rapid (10+0, 15+10), Classical (30+0, 30+20)
- **Зависимости:** Исправление синхронизации таймера (2.5 в refactoring-plan)
- **Трудзатраты:** M
- **Замечания по качеству:**
  - ⚠️ REST-эндпоинт `move.post.ts` НЕ обновляет часы — только WS-обработчик делает это. Два пути рассинхронизированы.
  - `clockSync` работает каждые 10 сек — допустимо, но возможен видимый drift.

### 1.2 rematch (Переигровка)
- **Статус:** Реализовано
- **Описание:** Кнопка "Rematch" после окончания игры. Отправка WS-сообщений `rematch_offer` / `rematch_accept` / `rematch_decline`. При accept — автоматическое создание новой игры с swapped цветами.
- **Трудзатраты:** M
- **Замечания по качеству:**
  - ⚠️ Нет server-side защиты от двойного offer — управляется только на клиенте через ref, что хрупко при перезагрузке страницы.

### 1.3 Прерывание игры (Abort)
- **Статус:** Реализовано
- **Описание:** Возможность прервать игру до первого хода без потери рейтинга. После первого хода — только resign.
- **Трудзатраты:** S

### 1.4 Email-верификация
- **Статус:** Не реализовано
- **Описание:** Подтверждение email при регистрации. Отправка письма с токеном, верификация при клике.
- **Трудзатраты:** L

### 1.5 Сброс пароля
- **Статус:** Реализовано
- **Описание:** Форма "Forgot password", отправка email с токеном для сброса. Токен живёт 1 час, одноразовый.
- **Трудзатраты:** L
- **Замечания по качеству:**
  - ⚠️ Ошибка отправки email глушится (catch) — пользователь видит "успех", но письмо может не прийти. Стоит добавить retry или уведомление о проблеме.
  - В остальном реализация корректна: анти-enumeration (`{ success: true }` всегда), одноразовый токен, expiry.

## Фаза 2: Социальные функции

### 2.1 Внутриигровой чат
- **Статус:** Частично реализовано
- **Описание:** Простой текстовый чат между игроками во время партии. WS-сообщения `chat`. ~~История сообщений сохраняется с игрой.~~
- **Трудзатраты:** M
- **Замечания по качеству:**
  - ❌ **История НЕ сохраняется.** Сообщения хранятся только в клиентском `ref`. При обновлении страницы чат пропадает. Нет таблицы `chat_messages` в БД.
  - ⚠️ Нет серверной санитизации сообщений — пользователи могут отправлять произвольный HTML/JS контент.
  - Реализована только real-time пересылка через WS.

### 2.2 Онлайн-статус пользователей
- **Статус:** Реализовано
- **Описание:** Показывать зелёный/серый индикатор рядом с username. Использовать `peerRegistry` для отслеживания подключённых пользователей. Отправлять `user_online` / `user_offline` через WS.
- **Трудзатраты:** M
- **Замечания по качеству:**
  - Реализация качественная: поддержка нескольких вкладок через `Set<WsPeer>`, корректные онлайн/оффлайн события только для друзей.
  - ⚠️ In-memory хранение — при рестарте сервера или multi-instance деплое статус потеряется (решается через Redis в 7.1).

### 2.3 Удаление аккаунта
- **Статус:** Не реализовано
- **Описание:** Кнопка "Delete account" в настройках профиля. Мягкое удаление (анонимизация данных) или полное. Обработка активных игр и друзей.
- **Трудзатраты:** M

### 2.4 Редактирование профиля
- **Статус:** Частично реализовано (баг)
- **Описание:** Добавить изменение username, avatar URL, био. API endpoint `PUT /api/users/me` расширен — username, bio, avatar.
- **Трудзатраты:** M
- **Замечания по качеству:**
  - ❌ **Отсутствует миграция для колонки `bio`.** В `schema.ts` поле `bio` определено, но миграция `00001_init.sql` его не создаёт, и последующих миграций для него нет. Обращение к `bio` вызовет ошибку БД.
  - ⚠️ Нет server-side проверки уникальности username перед update — при коллизии пользователь получит сырую ошибку БД.

## Фаза 3: Анализ и движок

### 3.1 Шкала оценки (Evaluation bar)
- **Статус:** Реализовано
- **Описание:** Визуальная полоса сбоку доски, показывающая оценку позиции. Обновляется после каждого хода. Использует Stockfish evaluation.
- **Трудзатраты:** M
- **Замечания по качеству:**
  - Хороший пул процессов Stockfish с очередью задач и таймаутами.
  - ⚠️ Эндпоинт `/api/engine/eval` не защищён авторизацией — любой может использовать анализ без логина (может быть намеренным решением).
  - Sigmoid-функция для визуальной шкалы корректна.

### 3.2 Анализ после игры (Post-game analysis)
- **Статус:** Типы `EngineEvaluation` и `AnalyzedMove` определены
- **Описание:** После окончания игры — автоматический анализ всех ходов. Показывать:
  - Лучший ход (Best move)
  - Ошибки/зевки (Blunder, Mistake, Inaccuracy)
  - Оценка позиции после каждого хода
  - Классификация ходов (!, ?, ??, ?!)
- **Трудзатраты:** XL

### 3.3 Multi-PV (несколько вариантов)
- **Статус:** Запланировано
- **Описание:** Показывать топ-3 линии Stockfish во время анализа. Отправлять `setoption name MultiPV value 3` движку.
- **Трудзатраты:** M

### 3.4 Подсказки (Hint arrows)
- **Статус:** Запланировано
- **Описание:** В режиме vs AI — кнопка "Hint", показывающая стрелку лучшего хода на доске. Использует `boardApi.drawArrow()`.
- **Трудзатраты:** S

### 3.5 Client-side Stockfish (WASM)
- **Статус:** Запланировано
- **Описание:** Загружать Stockfish WASM в браузере для мгновенного анализа без запросов к серверу. Экономит ресурсы сервера.
- **Трудзатраты:** L

## Фаза 4: Матчмейкинг

### 4.1 Быстрая игра (Quick Play / Matchmaking)
- **Статус:** Запланировано
- **Описание:** Система автоматического подбора соперника по рейтингу:
  1. Игрок выбирает time control и нажимает "Quick Play"
  2. Добавляется в очередь matchmaking
  3. Сервер подбирает соперника в диапазоне ±100 ELO (расширяется со временем)
  4. Автоматическое создание игры и уведомление обоих игроков
- **Компоненты:**
  - `server/utils/matchmaking.ts` — очередь и matching algorithm
  - WS-сообщения: `matchmaking:join`, `matchmaking:cancel`, `matchmaking:found`
  - UI: кнопка "Quick Play" на главной странице
- **Трудзатраты:** XL

### 4.2 Режим Spectator (Зритель)
- **Статус:** Типы определены, handler отсутствует
- **Описание:** Возможность наблюдать за чужой игрой в реальном времени:
  1. Ссылка для spectators (отдельный WS-канал или `game:watch`)
  2. Доска только для просмотра
  3. Счётчик зрителей
  4. Чат зрителей (опционально)
- **Трудзатраты:** L
- **Замечания:**
  - Определён тип `spectate` в `ClientMessage` и функция `spectateGame()` на клиенте.
  - ❌ На сервере в `_ws.ts` **нет обработчика** `case 'spectate'` — сообщение просто игнорируется.
  - Страница `game/[id].vue` — только просмотр завершённых игр через REST API, не live-трансляция.

## Критические баги и замечания по качеству

> Проблемы, найденные при аудите реализованных фич. Необходимо исправить перед дальнейшим развитием.

### BUG-1: Отсутствует миграция для колонки `bio`
- **Приоритет:** Высокий
- **Файлы:** `server/db/schema.ts` (определяет `bio`), `server/db/migrations/00001_init.sql` (нет `bio`)
- **Проблема:** Drizzle-схема определяет `bio: text().default('')`, но колонка не создана в БД. Все операции с `bio` (чтение/запись профиля) вызовут ошибку.
- **Решение:** Добавить миграцию `ALTER TABLE users ADD COLUMN bio TEXT NOT NULL DEFAULT '';`

### BUG-2: Чат не сохраняется в БД
- **Приоритет:** Средний
- **Файлы:** `server/routes/_ws.ts` (handler `chat`), `server/db/schema.ts`
- **Проблема:** Сообщения чата пересылаются real-time, но не персистятся. При обновлении страницы история теряется.
- **Решение:** Создать таблицу `chat_messages`, сохранять сообщения при пересылке, подгружать историю при `join`.

### BUG-3: REST `move.post.ts` не обновляет часы
- **Приоритет:** Средний
- **Файлы:** `server/api/games/[id]/move.post.ts`
- **Проблема:** REST-эндпоинт хода не обновляет `whiteTime`/`blackTime`, в отличие от WS-обработчика. Два пути расходятся.
- **Решение:** Либо убрать REST-эндпоинт (оставить только WS), либо добавить логику часов.

### BUG-4: Нет санитизации в чате
- **Приоритет:** Средний
- **Файлы:** `server/routes/_ws.ts` (handler `chat`)
- **Проблема:** Сообщения чата пересылаются без какой-либо санитизации. Потенциальный XSS.
- **Решение:** Добавить sanitization (например, `DOMPurify` на клиенте или strip-HTML на сервере).

### BUG-5: Утечка памяти в rate limiter
- **Приоритет:** Низкий
- **Файлы:** `server/utils/rateLimit.ts`
- **Проблема:** Записи в `Map` не удаляются активно — только при повторном обращении к ключу. При большом количестве уникальных IP Map растёт бесконечно.
- **Решение:** Добавить периодическую очистку просроченных записей (например, раз в минуту).
